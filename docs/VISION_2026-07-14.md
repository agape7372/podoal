## 0. 한 문장 총평

**엔진은 완성됐고, 계기판(계측)과 시운전(베타)을 켜는 사용자 액션 3종이 전체 대기열을 막고 있다.**
코드 관점에서 podoal은 드물게 성숙하다 — 프론트 모션/모달/제스처, 백엔드 인증가드/소유권경계/상수시간
cron, 데이터 핫패스 인덱스/Serializable 정합이 모두 프로덕션급이고 즉시 악용 가능한 Critical은 없다.
진짜 병목은 코드가 아니라 **신호**다: 리텐션 데이터가 0이고(PostHog 키 게이트), P1 완료정의의 유일
출처인 친구 베타 1라운드가 아직 안 돌아서, 다음 판단을 전부 시뮬레이션 페르소나에 의존하고 있다.

유일하게 "**지금 손대야 하는**" 코드 항목은 OAuth 자동병합 경화 하나였고 — 실OAuth 키 등록(대기 중
사용자 액션)이 되는 순간 잠재 계정탈취로 활성화되므로 — **이번 세션에 최소 경화 + 회귀 테스트로 처리**
했다(§4). 나머지는 게이트 없이 신뢰할 수 있는 안전한 위생/테스트 정리이거나, 사용자 결정·상위설계에
묶인 성장 카드로 깔끔히 갈린다.

## 2. 관통 테마 7

1. **제품은 코드가 아니라 '신호'에서 막혀 있다** — 3층 모두 프로덕션급, 다음 병목은 사용자 액션 3종+베타.
2. **OAuth 자동병합 취약점이 사용자게이트 '순서'를 강제한다** — 실키 등록 전에 반드시 경화(→ §4 처리).
3. **보안 임계 순수모듈·핸들러 테스트 공백 = 최고가치·최저비용 안전망** — DB 불필요, 즉시 가능(→ §4 착수).
4. **'안전한 위생 정리 층' vs '상위설계 층'의 선명한 분기 — 거버넌스 실증** — additive 정리는 즉시 신뢰 가능.
5. **감성 연출이 해자이자 커스터마이징의 '제약 조건'** — 색은 바꿔도 모션 타이밍·600ms 창은 불변.
6. **커스터마이징/개인화가 다음 성장 벡터, 전부 게이트 뒤 대기** — 인프라는 준비, 병목은 결정.
7. **스케일 천장은 인프라 티어에 묶인 '언제'의 문제** — 신호 없는 선제 착수는 과최적화, 설계문서부터.

## 4. 🔴 이번 세션 최우선 처리 — OAuth 계정병합 경화

### 발견 (HIGH, 검증 완료)
`api/auth/oauth/[provider]/callback/route.ts`의 `finalizeLogin`이, provider가 보고한 email이 기존 계정과
같으면 그 계정에 provider/providerId를 붙이고 JWT를 발급했다. 거부 가드가
`byEmail.provider && byEmail.provider !== providerTag` 뿐이라 **`provider === null`인 비밀번호 계정은
가드를 건너뛰고 자동 병합**됐다. `email_verified`도 확인하지 않으므로, 미검증 이메일을 허용하는 provider
(카카오/네이버)에서 공격자가 피해자의 가입 이메일을 자기 프로필 이메일로 설정해 OAuth를 완료하면 **피해자의
비밀번호 계정으로 로그인**된다.

### 왜 지금인가
현재는 `isRealOAuth=false`(전 provider 게스트 폴백)라 이 경로가 **휴면**이다. 그러나 대기 중 사용자 액션
("OAuth 실키 등록")이 되는 순간 즉시 활성화된다 → **실키 등록 전에 반드시 막아야 하는 하드 순서 제약**.

### 이번 조치 (커밋 `feat/fix`)
- `src/lib/oauth.ts`: 순수 판정 함수 `decideAccountMerge(existing, providerTag)` 추가 —
  `create`/`merge`/`reject`. **비밀번호 계정(provider=null)으로의 자동 병합을 거부**(`reason: 'password'`),
  같은 provider 재로그인만 merge, 다른 provider 소유 email은 기존대로 거부.
- 콜백을 이 판정에 배선. 거부 시 `oauth_email_taken_by_password`로 리다이렉트.
- `src/app/page.tsx`: 새 에러코드에 자연스러운 한국어 안내("이 이메일은 비밀번호로 가입한 계정이에요.
  비밀번호로 로그인해주세요.").
- 회귀 테스트 5건(`oauth.test.ts` — 병합 판정 4 + `isRealOAuth` 분기 1).
- **현 사용자 영향 0**: 게스트 폴백 해피패스 무변경. 바뀌는 건 (아직 비활성인) 실OAuth 경로의 안전한 기본뿐.

### 잔여 (사용자/상위 결정)
- **email_verified 플러밍**(방어 심화): `NormalizedUserInfo`에 provider별 검증 플래그를 실어 미검증이면
  생성/병합 금지. 3개 normalize + 타입 변경이라 별도 카드.
- **의도적 계정 연결 UX**: 로그인 상태에서 명시적으로 소셜 계정을 연결하는 기능(자동병합의 대체재).
  인증 계약이라 오너 정책 결정 선행.

### 함께 착수한 안전망 (즉시착수 tier, 게이트 없음)
- `cronAuth.test.ts`: `verifyCronAuth` 3케이스(미설정 false·정확일치만 true·길이불일치 no-throw). 3개 cron
  라우트의 유일 인증 관문이 리팩터 회귀로 조용히 깨지지 않게 고정.
- `src/lib/latestGuard.ts` + `useCachedApi` 역순 응답 가드(별도 커밋) — 동적 url 훅(friends/[id]·relay/[id])
  전환 시 늦게 온 이전 요청이 새 화면 state를 덮어쓰는 것 차단(07-14 리뷰가 지정한 "첫 코드 카드").

## 8. 커스터마이징 비전 (3층)

철칙: 사용자는 **색과 콘텐츠**를 바꾸고, **모션·타이밍·물리감각은 불변**으로 해자를 보존한다.

1. **스킨 층(R1, 무료·코스메틱)** — 품종 스킨(샤인머스캣·거봉·청포도)을 `GrapeSkin` 카탈로그 + `Board.skinId` +
   렌더러 팔레트 매핑으로. 재화 없이 색만 교체하므로 리스크 최소이자, 개인화 수요·리텐션 반응을 계측으로 관찰하는
   **첫 측정기**(A1 계측 가동 선행 필수). 이 데이터가 R2 가격·전환 설계의 근거.
2. **편의/접근성 층** — 다크모드(clay/warm/shadow-tint 토큰 라이트/다크 이층화 + grape 음영 재튜닝, 정본
   재정의라 별도 마라톤·설계문서 선행), 글자 크기 팩(설계 확정, 실기기 QA 게이트만).
3. **콘텐츠 개인화 층** — 커스텀 템플릿(신규 테이블), 보상 이미지 업로드(인프라 준비됨, 가족 세그 유료전환
   1순위), 공개 보드 링크(소셜/감성 4.50 세그의 바깥 유입 통로).

**통합 순서**: 계측(A1)이 켜져야 어느 커스터마이징이 실제 리텐션을 움직이는지 알 수 있으므로,
**스킨 R1이 계측 위에서 먼저 나가 나머지 개인화 우선순위를 데이터로 정하는** 순서가 이상적이다.
