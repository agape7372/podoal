## 2. 이번 패스 발견 이슈

### UX/UI·프론트엔드

1. **수정 — 공용 버튼의 암묵적 submit (중간)**: `ClayButton`의 버튼 분기가 `type`을 지정하지 않아 폼 내부의 취소·선택·재시도 버튼이 브라우저 기본값 `submit`으로 동작할 수 있었다. 공용 기본을 `button`으로 바꾸되 호출부가 명시한 `type="submit"`은 보존했다.
2. **수정 — 선물 친구 조회 실패를 빈 목록으로 오인 (중간)**: `GiftBoardModal`만 공용 `api()`를 우회해 `res.ok`를 검사하지 않았다. 401/500도 `{ friends: [] }`처럼 보이거나 파싱 실패가 비처리될 수 있었다. 오류 상태와 재시도를 분리했다.
3. **수정 — 선물 메시지 라벨 미연결 (접근성, 중간)**: 시각적 `<label>`과 textarea 사이에 `htmlFor`/`id` 연결이 없었다.
4. **발견·후속 — 동적 URL 캐시 경쟁 (중간)**: `useCachedApi(url)`은 동일 컴포넌트 인스턴스에서 URL이 바뀌고 이전 요청이 늦게 완료되면 이전 데이터를 새 화면 state에 쓸 수 있다. 핵심 조회축이므로 역순 응답 회귀 테스트를 먼저 만든 뒤 AbortController 또는 요청 세대 가드로 수정한다.
5. **관찰 — native button 명시성 (낮음)**: 여러 native `<button>`이 `type`을 생략한다. 현재 대부분 form 밖이지만 향후 래핑 변화에 취약하다. 이번에는 공용 버튼과 실제 수정 표면만 고치고 전역 기계 치환은 피했다.

### 백엔드·보안·성능

- 새 Critical/High는 확인되지 않았다. 보호 라우트/authZ, cron 경화, 핫 인덱스는 07-13 결과와 일치한다.
- 선물 모달의 실패 오인은 서버 결함이 아니라 클라이언트 HTTP 처리 불일치였다. 공용 `ApiError` 경로로 합쳤다.
- 구조화 로깅·SSE·CSP는 외부 도구/아키텍처 결정이 필요한 기존 보류 항목이라 코드 변경하지 않았다.

### 접근성

- 공용 `Modal`은 dialog 의미론, 포커스 트랩, Escape, 초기 포커스와 복원을 이미 제공한다.
- 이번 수정으로 선물 메시지 입력의 accessible name을 확정하고 조회 오류를 `role="alert"`로 알린다. 기존 전역 포커스 링·reduced-motion 규칙은 중복 추가하지 않았다.

## 3. 실제 구현과 이유

- `src/components/ClayButton.tsx`: 기본 `type="button"`; 명시적 submit은 그대로 전달. 한 곳에서 폼 오동작 계열을 예방하는 최소 변경이다.

- 스키마·API 응답 계약·시크릿·배포 설정은 변경하지 않았다.
