## 1. 방법론 (차기 감사 재사용 — PLAYBOOK에서 링크)

**하위모델 탐색 재검증 게이트가 이 감사의 1원칙.** 사전 Explore 보고의 "High 2건"이 재검증에서 전부 무효(§3)였다. 모든 외부 보고는 코드+실행으로 페이블이 재확인.

영역: A 백엔드 라우트 전수(AuthZ 우회·상태머신·입력폭탄·정보누수·자원남용·dev게이트·CSRF) / B 프론트(컨벤션 grep + 상태 매트릭스) / C 연계 매트릭스 / D 실동작 / E 푸시·크론 / F 문구(선물·공유·메시지) / G 문서 정합.

## 2. 발견 (PA-NNN)

### PA-006 · Med · 보안(남용) · `src/app/api/messages/route.ts:89` · 배정 sonnet
**비친구에게 메시지 전송 가능.** POST /api/messages는 수신자 존재만 확인하고 친구 관계를 검증하지 않는다. podoal은 친구 기반 소셜 모델인데 임의 userId에 응원/축하 메시지를 보낼 수 있다(원치 않는 메시지 벡터).
- 재현(실측 201): dev 유저 → 별도 register 유저(비친구) id로 `POST /api/messages {receiverId, content, type:'cheer'}` → **201 생성됨**.
- 완화 현존: rate-limit 20/분, 500자 캡, 자기송신 차단.
- 수정: accepted Friendship(양방향 OR) 검증 추가, 없으면 403 한국어. **주의**: 선물/깜짝선물/포도동 알림은 라우트가 아니라 `prisma.message.create` 직접 호출(gift/route.ts:88, plant-gift:80)이라 이 게이트에 영향 없음 — 반드시 POST 라우트에만 적용.

### PA-008 · Med · 연계 단절(사용자 중점) · `src/app/(app)/messages/page.tsx:138` · 배정 sonnet
**선물·축하 메시지에서 해당 포도판으로 갈 수 없다.** MessageInfo는 `boardId`·`sender.id`를 보유하는데, 메시지 카드 탭은 읽음 처리만 한다. 선물 메시지("포도판을 선물했어요")를 받아도 그 보드로 못 간다 — 선물 흐름의 핵심 연결이 끊김.
- 수정: `boardId`가 있으면 카드/전용 버튼으로 `/board/{boardId}` 이동. 읽음 처리와 이동을 함께(이동 전 markRead). 삭제 버튼 클릭 버블링은 기존 형제 구조 유지. sender→프로필 링크는 PA-006 게이트 적용 후 안전(친구만 발신).

### PA-004 · Med · 멱등성 · `src/app/api/boards/[id]/gift/route.ts` · 배정 sonnet
**같은 보드를 같은 친구에게 반복 선물 시 매번 사본 생성**(서버 멱등성/쿨다운 없음). 원본 보드는 여러 친구에게 선물 가능이 의도지만, 동일 (원본, 수신자) 쌍의 중복 사본은 친구 홈을 오염시킬 수 있다.
- 수정: 동일 senderId·giftedFromId(원본)·giftedToId 조합의 미개봉(giftOpenedAt=null) 사본이 이미 있으면 409 "이미 선물했어요"로 차단. (개봉 후 재선물은 허용 — 판단.)

### PA-002 · Low · 입력검증 · `src/app/api/boards/[id]/stickers/route.ts:16` · 배정 haiku
**비정수 position이 500 유발.** `position: "0"`(문자열) → 범위검사 통과 → Prisma Int 조회 타입오류 → 500(실측). float(1.5)는 잘못된 409를 냄.
- 수정: `if (!Number.isInteger(position)) return authResponse('잘못된 칸이에요', 400);`를 범위검사 앞에. (plant-gift·rewards는 이미 Number.isInteger 가드 보유 — 동일 패턴.)

### PA-003 · Low · 입력검증 · `src/app/api/boards/[id]/gift/route.ts:18` · 배정 haiku
**비문자열 friendId가 500 유발.** `friendId:{a:1}` → `!friendId` 통과 → Prisma 조회 타입오류 → 500(실측).
- 수정: `if (typeof friendId !== 'string' || !friendId) return authResponse('Missing required field: friendId', 400);`

### PA-007 · Low · 자원남용 · `src/app/api/notifications/reminders/route.ts:34` · 배정 sonnet
**리마인더 message 길이 무제한 + 사용자당 개수 제한 없음.** 5000자 메시지 저장 성공(실측), 리마인더 무한 생성 가능. 크론이 매 발송마다 이 문자열을 푸시 body로 실어 나른다.
- 수정: message 길이 캡(예 200자, 초과 400), 사용자당 리마인더 개수 캡(예 50개, 초과 400 "리마인더가 너무 많아요").

### PA-009 · Low · 입력 일관성 · `src/app/api/boards/[id]/gift/route.ts:16` · 배정 haiku
선물 메시지 서버 `.slice(0,500)` vs 클라 `GiftBoardModal` `maxLength={200}`. 표시 계약은 200자 — 서버 캡을 200으로 통일(방어적, 무해).

### PA-005 · 문서 · `CLAUDE.md` · 배정 fable (Phase 5)
CLAUDE.md stale 조항 다수: 홈 필터 "3탭" 기술(실제 4탭 — 전체/진행/완료/수확), 스와이프 수확·`harvestedAt`·`cellarNote`(소믈리에 노트)·통합 알림 피드(`/api/notifications`) 미기술. + 데이터 레이어 "do not modify" → 게이트 방식으로 현행화. Phase 5에서 PRINCIPLES 신설과 함께 일괄.
